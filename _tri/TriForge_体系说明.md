# TriForge 体系说明

Tri（三线）+ Forge（锻造） — 三条线共同锻造一个产品。

---

## 一、概述

TriForge 是一套面向"一人团队"的 AI Agent 多角色协作框架。它将完整的产品/设计/技术团队分工抽象为 15 个 Agent 角色 + 1 套共识对冲机制，通过结构化的提示词和文件流转，让一个人借助 AI 完成从需求到上线的全流程。

### 解决的问题

- 独立开发者缺乏团队级决策质量（产品判断力、设计还原度、技术可行性评估）
- 传统多 Agent 系统是单向流水线，缺乏交叉校验，信息损耗逐级放大
- 冲突在开发过程中意外暴露，返工成本高

### 核心思想

将产品/设计/技术三线的信息不对称和目标冲突从"意外"变成"流程的一部分"——在开发的决策阶段（而非执行阶段）强制暴露和解决冲突。

---

## 二、三层协作模型

```
┌─────────────────────────────────────────────────────────────────────┐
│                    你（用户）—— 最终决策者                          │
│                      PRD 定稿 / 设计定稿 / 技术栈确认              │
│                      变更裁定 / 对冲仲裁 / 上线决策                │
└────────────────────────────┬────────────────────────────────────────┘
                             │ 对话
                             ▼
┌─────────────────────────────────────────────────────────────────────┐
│              总理 Agent（战略层 · 全局窗口 · 自定义提示词）          │
│                                                                     │
│  角色：你的秘书/总项目经理/总参谋。你只跟总理对话，                 │
│       就能了解全局进度、做出关键决策。                              │
│                                                                     │
│  职责：任务拆解 → 进度总览 → 告知你下一步该找谁 → 变更处理        │
│        → 临时任务判断 → 项目复盘                                   │
│                                                                     │
│  ⚠️ 不自作主张修改产出，异常时停止报告，等待你决策                 │
└────────────────────────────┬────────────────────────────────────────┘
                             │ 你根据总理指引，通知 SOLO 执行
                             ▼
┌─────────────────────────────────────────────────────────────────────┐
│              SOLO（执行层 · Trae 原生传令官）                        │
│                                                                     │
│  角色：调度 Sub Agent，多任务并行，独立上下文                        │
│  能力：自动调用开启"可被其他智能体调用"的 Sub Agent                  │
│        Sub Agent 之间不能互调，只能由 SOLO 统一调度                  │
└─────────────────────────────────────────────────────────────────────┘
                             │
          ┌──────────────────┼──────────────────┐
          ▼                  ▼                  ▼
     product-line/      design-line/       tech-line/
     （6 份）           （1 份）            （5 份）
     产品经理 → PRD    设计师             技术经理 → 测试
```

### 体系向导

**指南 Agent** 独立于三层模型之外，与总理 Agent 同级。不参与项目执行，只回答关于 TriForge 体系如何运行的问题（流程、角色、规则、文件结构）。

## 三、核心理念

### 三线分离

| 线 | 关注的问题 | 负责人 |
|----|-----------|--------|
| 产品线 | 做什么、为什么做 | 产品经理 Agent |
| 设计线 | 怎么呈现 | 设计师 Agent |
| 技术线 | 能不能做、怎么做 | 技术经理 Agent |

### 冲突前置

三线天然存在利益冲突。该体系通过**三个共识/对冲节点**在开发执行前强制暴露这些冲突，完成交叉校验。

```
主控层（战略层）
├── 总理 Agent      — 秘书/总项目经理/总参谋，你的全局窗口
└── 指南 Agent      — 体系向导（平行，不参与执行）

产品线（产品经理 Agent 统筹）
├── 需求分析师      ① → 01_需求池.md
├── 竞品分析师      ② → 02_竞品分析.md  [需 DuckDuckGo MCP]
├── 用户研究员      ③ → 03_用户画像.md
├── 功能架构师      ④ → 04~06 定位/链路/逻辑
└── PRD 撰写师      ⑤ → 07~08 信息架构/PRD

设计线（设计师 Agent 一人全包）
  需求解析 → 设计系统 → 交互稿 → 视觉稿 → 设计规范.json

技术线（技术经理 Agent 统筹）
├── 前端工程师      — frontend/ 代码实现
├── 后端工程师      — backend/ 代码实现 + 部署指导
├── 数据库工程师    — database/ SQL 脚本
└── 测试 Agent      — 验收报告（开发完成后启用）

共识与对冲节点
├── 产品-设计共识  — PRD + 设计系统对齐
├── 产品-技术对冲  — PRD + 技术方案讨论 → 技术栈决策
└── 设计-技术共识  — 设计规范 + 技术方案对齐

资产层
└── 模板沉淀机制    — 抽取可复用模板，反哺后续项目
```

## 八、完整工作流程

### Step 1 — 启动

```
你输入项目目标
→ 总理读取 PROJECT_BRIEF.md（手动确认已加载）
→ 总理拆解任务 → 写入 TASK_QUEUE.md
→ 总理告诉你：启动产品线，先找 @需求分析师
```

### Step 2 — 产品线串行（每步回总理确认）

```
① 需求分析师 → 01_需求池.md → 汇报 → 你确认 → 总理指引下一步
② 竞品分析师 → 02_竞品分析.md → 汇报 → 你确认 → 总理指引下一步
③ 用户研究员 → 03_用户画像.md → 汇报 → 你确认 → 总理指引下一步
④ 功能架构师 → 04~06 定位/链路/逻辑 → 汇报 → 你确认 → 总理指引下一步
⑤ PRD 撰写师 → 07~08 信息架构/PRD → 汇报
→ 产品经理汇总质检 → 【你决策：PRD 定稿】
```

### Step 3 — 设计与技术并行 + 共识对冲

```
PRD 定稿后：
  设计师启动设计线（设计系统→交互→视觉→规范）
  技术经理启动技术线（评审→方案→排期）

共识与对冲节点：
├─ 产品-设计共识：PRD + 设计系统对齐
├─ 产品-技术对冲：PRD + 技术方案 → 【你决策：技术栈】
└─ 设计-技术共识：设计规范 + 技术方案对齐
└─ 分歧升级 → 【你决策：仲裁】
```

### Step 4 — 技术执行

```
前端 → 后端 → 数据库 → 联调 → 测试 Agent 验收
→ 变更请求 → 回总理 → 【你决策：变更裁定】
→ 后端产出部署说明（mvn package + 启动方式）
```

### Step 5 — 验收与复盘

```
测试 → 验收报告
→ 【你决策：上线 / 迭代】

总理发起复盘（上线或迭代后）：
  ① 这次什么东西做得好？
  ② 什么地方踩坑了？
  ③ 下次怎么改进？
→ 整理为复盘记录存入 PROJECT_MEMORY.md
```

### Step 6 — 资产沉淀

```
抽取可复用模板 → 存入 templates/
→ 反哺后续项目
```

## 九、共识与对冲机制

### 三个节点

| 节点 | 参与方 | 触发时机 | 产出物 |
|------|--------|---------|--------|
| 产品-设计共识 | 产品经理 + 设计师 | PRD 页面结构确认后 | 可设计性确认书 + 设计排期 |
| 产品-技术对冲 | 产品经理 + 技术经理 | 技术评审时 | 可行性结论 + 替代方案 |
| 设计-技术共识 | 设计师 + 技术经理 | 设计稿交付后、开发前 | 可实现性确认 + 还原标准 |

### 分歧升级路径

```
Agent 分歧 → 各自陈述立场 → 能否达成一致？
  ├─ 能 → 共识达成，继续
  └─ 不能 → 总理汇总 → 汇报你 → 【你决策】→ 写入 TECH_DECISIONS.md
```

## 十一、运行模式

### 项目启动

1. `npx triforge init my-project` 或复制 `_tri/` + `templates/` 到项目根目录
2. 填写 `PROJECT_BRIEF.md`：产品背景、定位、目标用户、约束
3. 填写 `环境与约定说明.md`：你的开发环境配置
4. 在 AI 编程工具中创建各 Agent（粘贴提示词，开启"可被其他智能体调用"）

### 运行时协作规则

- **你只跟总理 Agent 对话**，了解全局进度
- **总理告诉你下一步需要找谁**，你通过 **SOLO** 下达执行指令
- **Sub Agent 完成任务后向你汇报**，你回到总理确认
- **Sub Agent 之间不互调**，只能由 SOLO 统一调度
- **异常时暂停**，向总理汇报，等待你决策

### 边界情况已覆盖

| 场景 | 处理方式 |
|------|---------|
| 项目中断后恢复 | 总理启动时读文件对齐进度 |
| 跳过步骤提前唤醒 | Sub Agent 先检查上游依赖 |
| 手动修改产出文件 | 版本号 + 责任 Agent 同步 |
| 忘写状态同步 | 总理对比时间戳做审计 |
| 跨线绕路（直接找 Sub Agent） | Sub Agent 同步文件，总理读文件对齐 |
| 临时任务 | 总理判断是咨询还是变更 |
| 突发故障 | 技术经理诊断给方案 |
| 决策回退 | 产品经理评缺失 + 技术经理评成本 → 你拍板 |

## 十三、技术栈选型规则

1. **技术经理 Agent** 在技术评审阶段评估并推荐技术栈方案
2. 推荐方案写入 `02_技术方案.md` 的"技术选型"章节
3. **用户** 最终确认技术栈
4. 确认后写入 `TECH_DECISIONS.md`

## 十四、用户必须决策的节点

| # | 决策点 | 触发时机 |
|---|--------|---------|
| 1 | PRD 定稿 | 产品线产出后 |
| 2 | 设计定稿 | 设计线产出后 |
