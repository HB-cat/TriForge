# TriForge — 全局规则文档

> 本文档定义 TriForge 体系的运行规则，包括共识与对冲机制的判断标准、分歧升级路径、决策记录规范以及边界场景声明。
> 所有 Agent 的提示词中涉及的共识/对冲触发条件，判断标准统一引用本文档。

---

## 一、概述

### 体系运行模型

TriForge 采用三层协作模型：

- **你（用户）** — 最终决策者
- **总理 Agent（自定义提示词）** — 战略层，你的秘书/总项目经理/总参谋，只负责全局视角和决策入口
- **SOLO（Trae 原生能力）** — 执行层传令官，负责调度 Sub Agent，每个 Sub Agent 拥有独立上下文

> 你不直接调度 Sub Agent。你通过总理 Agent 了解全局，总理告诉你下一步该找谁，你通过 SOLO 下达执行指令。Sub Agent 之间不能相互调用，只能由 SOLO 统一调度。

### 共识机制定位

共识与对冲机制是 TriForge 的核心设计。在三线（产品/设计/技术）协作过程中，于关键节点设置交叉校验，确保所有冲突在技术执行前被识别和解决。

## 二、三个节点

### 节点 1：产品-设计共识

| 属性 | 内容 |
|------|------|
| 参与方 | 产品经理 Agent + 设计师 Agent |
| 触发时机 | PRD 页面结构确认后 |
| 产出物 | 可设计性确认书 + 设计排期 |

**判断标准：**
- PRD 中的每个页面/模块是否有对应的设计系统组件或新增组件说明
- 交互流程是否覆盖了 PRD 中定义的全部用户场景
- 双方对齐后 = 通过
- 存在分歧且无法达成一致 = 不通过 → 升级

### 节点 2：产品-技术对冲

| 属性 | 内容 |
|------|------|
| 参与方 | 产品经理 Agent + 技术经理 Agent |
| 触发时机 | 技术评审时 |
| 产出物 | 可行性结论 + 不可行项替代方案 + 决策记录 |

**判断标准：**
- 每项功能需求是否有明确的技术实现路径
- 识别出不可行项时，技术经理必须提供替代方案（不能只说"做不了"）
- 产品经理评估替代方案是否能满足原始需求
- 双方对齐后 = 通过
- 存在分歧 = 不通过 → 升级

**技术栈选型在此节点完成：** 技术经理推荐方案，用户最终确认。

### 节点 3：设计-技术共识

| 属性 | 内容 |
|------|------|
| 参与方 | 设计师 Agent + 技术经理 Agent |
| 触发时机 | 设计稿交付后、开发前 |
| 产出物 | 可实现性确认 + 技术约束反馈 + 设计还原标准 |

**判断标准：**
- 每个设计元素是否有对应的技术实现方案
- 设计规范中的 Design Tokens 是否可转化为前端 CSS 变量或组件 Props
- 存在技术约束无法还原的设计 = 不通过 → 升级（设计师调整设计或技术经理提供变通方案）
- 对齐后 = 通过

## 三、分歧升级路径

```
Agent 间分歧
    ↓
双方各自陈述立场和依据（写入 TECH_DECISIONS.md）
    ↓
┌─ 能否达成一致？
├─ 能 → 共识达成，继续流程
└─ 不能 → 升级
        ↓
   总理 Agent 汇总分歧
        ↓
   向用户（你）汇报双方立场
        ↓
   【用户决策】
        ↓
   结果写入 TECH_DECISIONS.md，继续流程
```

### 升级触发条件
- 双方 Agent 经过两轮沟通仍无法达成一致
- 某方 Agent 认为对方方案存在不可接受的风险
- 涉及技术栈选型时，技术经理推荐的方案与产品经理预期不符（需用户定夺）

## 四、决策记录格式

所有共识/对冲/仲裁结果必须写入 `_tri/memory/TECH_DECISIONS.md`，格式如下：

```markdown
## 决策记录 YYYY-MM-DD-HHmm

### 节点
产品-技术对冲 / 产品-设计共识 / 设计-技术共识 / 用户仲裁

### 参与方
- [Agent 名称]：[立场摘要]
- [Agent 名称]：[立场摘要]

### 分歧点
[具体分歧描述]

### 结论
[最终决定]

### 决策人
[达成共识 / 用户]

### 影响
[对项目的影响说明]
```

### 记录规范
- 每次决策独立一条，按时间排序
- 每条记录以日期时间戳开头（YYYY-MM-DD-HHmm）
- 不可删除或修改历史记录，只能追加新记录

## 五、各 Agent 引用说明

| Agent | 涉及的共识/对冲节点 |
|-------|-------------------|
| 总理 Agent | 监督所有节点是否被触发，处理升级请求 |
| 产品经理 Agent | 产品-设计共识、产品-技术对冲 |
| 设计师 Agent | 产品-设计共识、设计-技术共识 |
| 技术经理 Agent | 产品-技术对冲、设计-技术共识 |
| 其他 Agent | 不直接参与共识/对冲，但产出物作为判断依据 |

## 七、边界场景声明

以下场景是流程设计中有意覆盖的边界情况，归因说明引用对应 Agent 提示词中的具体处理规则。

| # | 场景 | 处理原则 | 具体执行位置 |
|---|------|---------|-------------|
| 1 | **项目中恢复** — 项目中断一段时间后恢复，对话上下文已过期 | 恢复时需确认当前进度状态，以文件记录为准而非对话记忆 | 总理 Agent → 工作流程第 1 步"恢复检查" |
| 2 | **跳过步骤提前唤醒** — 上游文件未完成时直接启动下游 Agent | 下游 Agent 应先检查上游依赖是否完备，不全时向用户说明风险，用户明确指令后执行 | 每个 Sub Agent → 行为准则"依赖检查" |
| 3 | **你手动改产出文件** — 用户在 IDE 中直接修改了 Agent 的产出文件 | 产出文件使用版本命名，最后维护该文件的 Agent 在下次被唤醒时检查版本变更并同步 | 每个 Sub Agent → 行为准则"版本同步" |
| 4 | **Sub Agent 忘写同步** — Agent 完成任务后未更新 PROJECT_MEMORY.md | 主控应主动校验文件时间戳与记录的一致性，发现偏差时向用户确认实际进度 | 总理 Agent → 工作流程第 2 步"同步审计" |
