> 编写目标：为 Trae 平台编写 13 个 Agent 提示词 + 1 份全局规则文档  
> 编写方式：写一份确认一份，不批量输出  
> 更新日期：2026-06-28

## 一、体系架构（三层模型）

```
你（用户）—— 最终决策者
    │ 对话
    ▼
┌─────────────────────────────────────────────┐
│  总理 Agent（自定义智能体）                    │
│  战略层 · 全局窗口 · 进度总览 · 决策入口      │
│  提示词由我们编写，完全可控                    │
│  你只跟它对话，看完全局再做决策                │
└──────────────────┬──────────────────────────┘
                   │ 你根据总理指引，通知 SOLO
                   ▼
┌─────────────────────────────────────────────┐
│  SOLO（Trae 原生能力）                       │
│  执行层 · 传令官 · 多任务并行 · 独立上下文    │
│  自动调度 Sub Agent，干完活回来报告           │
└──────┬──────┬──────┬──────┬──────┬─────────┘
       │      │      │      │      │
       ▼      ▼      ▼      ▼      ▼
   需求   竞品   用户  功能   PRD   ...（Sub Agent）
   分析师  分析师  研究员 架构师  撰写师
```

### 三层各司其职

| 层 | 角色 | 做什么 | 提示词谁写 |
|----|------|--------|-----------|
| **你** | 决策者 | 做决策、确认产出、仲裁分歧 | — |
| **总理 Agent** | 战略层 | 全局进度、决策入口、告诉你要找谁 | ✅ 我们 |
| **SOLO** | 执行层 | 传令官，自动调度 Sub Agent | ❌ Trae 内置，不动 |
| **Sub Agent** | 执行单元 | 干具体活（写文档、写代码） | ✅ 我们 |

## 三、统一提示词结构

每份提示词使用以下 9 个 section：

```
# [Agent 名称] — 提示词

## 元信息        → 名称、所属层级、上级 Agent、配置优先级、角色定位（一句话）
## 角色定位       → 详细描述角色定位和存在价值
## 核心职责       → 3-5 条核心职责
## 行为准则       → 工作范围约束、协作规范、关键原则
## 输入依赖       → 输入名称 / 来源 Agent / 格式要求
## 输出交付       → 输出名称 / 目标 Agent / 格式要求
## 工作流程       → 接收→执行→输出 步骤
## 协作接口       → 向上汇报 / 向下分发 / 平行协作
## 版本记录       → 版本 / 日期 / 修改内容
```

## 五、关键设计决策

| 决策项 | 结论 |
|--------|------|
| 总控 Agent | 总理 Agent（自定义提示词），不做在 SOLO 里（SOLO 提示词不可改） |
| 调度机制 | 你跟总理对话 → 总理告诉你下一步 → 你通过 SOLO 调度 Sub Agent |
| Sub Agent 互通 | 不能互调，只能由 SOLO 调用。每个 Sub Agent 独立上下文 |
| 共识与对冲机制 | 触发条件写入 Agent 提示词，判断标准放入全局规则文档 |
| Project Brief | 提示词中显式写入读取路径 `_tri/memory/PROJECT_BRIEF.md`，对话开始时手动确认已加载 |
| 环境与约定说明 | 提示词中引用 `_tri/memory/环境与约定说明.md`，确保生成代码符合 IDE 规范 |
| 技术栈选型 | 技术经理 Agent 在技术评审阶段评估推荐，用户最终确认 |
| 编写顺序 | 全局规则文档优先写，再按配置顺序逐轮编写 |
| 编写方式 | 写一份确认一份，不批量输出 |
| 通信方式 | 总理负责汇报进度和调度建议，你通过 SOLO 下达执行指令（非自动流转） |
| 跨线绕路 | 用户直接找 Sub Agent → Sub Agent 同步到文件 → 总理读文件对齐（不依赖对话记忆） |
| 异常处理 | 任何异常立即停止，向用户报告，等待决策 |

## 四、额外场景预案

| # | 场景 | 风险 | 预案 | 写入谁的提示词 |
|---|------|------|------|--------------|
| 1 | **项目中恢复** — 中断一周后恢复，对话已过期 | 总理和 Sub Agent 都丢失进度 | 总理工作流程加"启动恢复检查"：对话开始时先读 TASK_QUEUE.md，向用户汇报当前进度并确认是否最新状态 | 总理 Agent |
| 2 | **跳过步骤提前唤醒** — PRD 没定稿直接找设计师 | 下游 Agent 基于草稿做无用功 | 每个 Sub Agent 行为准则加"依赖检查"：工作前确认上游依赖文件存在且非草稿。缺失时向用户说明缺少什么。用户明确指令后依令执行 | 每个 Sub Agent |
| 3 | **你手动改产出文件** — 你在 IDE 里直接改了 PRD | 各 Agent 基于旧版本工作 | 产出文件使用版本号（如 `08_PRD_v2.md`），Agent 读取最新版本；最后维护该文件的 Agent 下次被唤醒时检查版本变更 | 每个 Sub Agent + 总理 |
| 4 | **Sub Agent 忘写同步** — 干完活忘了更新 PROJECT_MEMORY.md | 总理进度信息过时 | 总理每次被找你时执行"同步审计"：比较 output/ 下文件的修改时间戳与 PROJECT_MEMORY.md 记录，发现不一致时向你确认实际进度 | 总理 Agent |

## 六、用户必须决策的节点

| # | 决策点 | 触发时机 | 说明 |
|---|--------|---------|------|
| 1 | PRD 定稿 | 产品线产出后 | 确认功能范围、产品做什么 |
| 2 | 设计定稿 | 设计线产出后 | 确认交互方式和视觉效果是否符合预期 |
| 3 | 技术栈选择 | 技术评审时 | 技术经理推荐方案，你确认技术体系 |
| 4 | 变更请求裁定 | 项目进行中随时 | 新增/修改功能，你来定做不做、何时做 |
| 5 | 共识/对冲仲裁 | Agent 分歧无法解决时 | 双方升级到你，做出最终判断 |
| 6 | 上线/迭代决策 | 验收完成后 | 发版上线，或打回继续迭代 |

决策结果写入对应记忆文件：
- PRD 定稿 → `PROJECT_MEMORY.md`
- 技术决策 → `TECH_DECISIONS.md`
- 变更请求 → `TASK_QUEUE.md`

## 九、各 Agent 提示词编写要点

### 全局规则文档

编写要点：
- 三个共识/对冲节点的触发条件定义
- 每个节点的判断标准（什么算通过、什么算不通过）
- 分歧升级路径（提出方 → 总理 → 用户决策）
- 决策记录写入 `TECH_DECISIONS.md` 的格式规范
- 各 Agent 在提示词中写明触发条件，判断标准统一引用此文档

### 总理 Agent（战略层 · 全局窗口）

| 维度 | 要点 |
|------|------|
| 角色定位 | 你的全局窗口和决策入口。你不直接找其他 Agent，只跟总理对话 |
| 核心职责 | 全局任务拆解、进度总览与汇报、跨线冲突识别、关键节点质量检查、变更请求处理、告诉你下一步该找谁 |
| 行为准则 | 不自作主张修改产品/设计/技术线的产出；异常时先停止再向你报告 |
| 协作方式 | 你不直接调度 Sub Agent，总理告诉你"下一步需要 @竞品分析师 做竞品分析"，你通过 SOLO 下达指令 |
| 特殊要求 | 判断"轻量模式"——小型项目可跳过部分子 Agent |

### 产品经理 Agent

| 维度 | 要点 |
|------|------|
| 核心职责 | 承接总理的任务，调度 5 个子 Agent，汇总产出，把控 PRD 质量 |
| 行为准则 | 工作区限于 `_tri/output/product/`，不跨区写入 |
| 协作接口 | 作为 Sub Agent 被 SOLO 调用，产出后通过总理向你汇报，平行参与共识节点 |

## 十、完整工作流程

```
Step 1 — 启动
  你：输入项目目标
  → 你跟总理 Agent 对话
  → 总理读取 PROJECT_BRIEF.md（手动确认已加载）
  → 总理读取 环境与约定说明.md
  → 总理拆解任务 → 写入 TASK_QUEUE.md
  → 总理告诉你：启动产品线，先找 @需求分析师

Step 2 — 产品线串行（你+总理+SOLO 配合）
  你 → 根据总理指引 → 通过 SOLO 唤醒需求分析师
  → 需求分析师产出 01_需求池.md → 汇报给你
  → 你回到总理 → 总理看过后确认 → 告诉你下一步

  → 你 → 通过 SOLO 唤醒竞品分析师[需 DuckDuckGo MCP]
  → 竞品分析师产出 02_竞品分析.md → 汇报给你
  → 你回到总理 → 总理看过后确认 → 告诉你下一步

  → 你 → 通过 SOLO 唤醒用户研究员
  → ...（以此类推，每步走完回总理确认）

  → PRD 撰写师产出 07_信息架构.md + 08_PRD.md → 汇报给你
  → 产品经理 Agent 汇总质检
  → 回到总理 → 【你决策：PRD 定稿】

Step 3 — 设计与技术并行
  PRD 定稿后，总理告诉你：
  "下一步需要 @设计师 启动设计线，@技术经理 启动技术线"

  你 → 通过 SOLO 分别唤醒设计师和技术经理（可并行）

  共识与对冲节点：
  ├─ 【产品-设计共识】PRD + 设计系统对齐
  ├─ 【产品-技术对冲】PRD + 技术方案讨论 → 【你决策：技术栈】
  └─ 【设计-技术共识】设计规范 + 技术方案对齐

  → 分歧无法解决 → 升级到【你决策：共识/对冲仲裁】

Step 4 — 技术执行（按需）
  共识通过后，总理告诉你安排工程师
  你 → 通过 SOLO 唤醒 前端/后端/数据库工程师

  → 项目进行中出现变更 → 回总理 → 【你决策：变更请求裁定】

Step 5 — 验收
  你 → 通过 SOLO 唤醒测试 Agent
  → 测试用例 + 验收报告
  → 设计师 → 设计还原度检查
  → 产品经理 → 功能验收
  → 【你决策：上线/迭代】

Step 6 — 资产沉淀
  抽取可复用模板 → 存入 templates/
  反哺后续项目
```

### 一句话总结

> **你跟总理对话了解全局 → 总理告诉你下一步 → 你通过 SOLO 执行 → 干完活回总理继续。**

## 十一、验证方式

每份提示词写完后，按以下标准检查：

1. **内容覆盖** — 是否覆盖了「各 Agent 提示词编写要点」中列出的全部内容
2. **协作接口对齐** — 检查 A Agent 的输出交付是否匹配 B Agent 的输入依赖，文件路径是否一致
3. **共识引用正确** — 涉及共识/对冲节点的 Agent，触发条件是否在提示词中写明，判断标准是否引用全局规则文档
4. **工作区约束** — 行为准则中是否明确写明了该 Agent 的工作区范围
5. **路径一致性** — 所有输入输出路径是否与文件结构约定一致
6. **异常处理** — 行为准则中是否声明了异常处理方式（停止→报告→等待决策）
